The Performance Advisor analyzes your Convex application and provides intelligent recommendations to improve query performance, optimize indexes, and enhance overall system health.

## Overview

Performance Advisor uses AI and static analysis to identify bottlenecks, suggest optimizations, and help you build faster, more efficient Convex applications.

<Tip>
  Performance Advisor continuously monitors your application and updates recommendations in real-time.
</Tip>

## Index Optimization

### Missing Index Detection

Automatically identifies queries that would benefit from indexes:

<Tabs>
  <Tab title="Recommendation">
    **Index Suggestion**
    
    ```typescript
    // ⚠️ Detected Issue
    // Query in: api/messages:listByUser
    // Pattern: Filtering by userId without index
    
    await ctx.db.query("messages")
      .filter(q => q.eq(q.field("userId"), userId))
      .collect();
    
    // 🚀 Recommended Fix
    // Add index in convex/schema.ts:
    
    messages: defineTable({
      userId: v.id("users"),
      text: v.string(),
      // ... other fields
    }).index("by_user", ["userId"]),
    ```
    
    **Impact**: Query time reduced from ~200ms to ~5ms (40x faster)
  </Tab>
  
  <Tab title="Auto-Fix">
    **One-Click Apply**
    
    <Steps>
      <Step title="Review suggestion">
        Check the recommended index configuration
      </Step>
      <Step title="Click 'Apply Fix'">
        Automatically adds index to schema file
      </Step>
      <Step title="Deploy changes">
        Push updated schema to deployment
      </Step>
      <Step title="Verify improvement">
        Monitor query performance after index creation
      </Step>
    </Steps>
    
    <Note>
      Auto-fix modifies your schema.ts file. Review changes before deploying.
    </Note>
  </Tab>
  
  <Tab title="Compound Indexes">
    **Multi-Field Indexes**
    
    For queries filtering on multiple fields:
    
    ```typescript
    // Query pattern detected:
    await ctx.db.query("messages")
      .filter(q => 
        q.and(
          q.eq(q.field("userId"), userId),
          q.eq(q.field("channel"), channel)
        )
      )
      .collect();
    
    // Recommended compound index:
    messages: defineTable({
      // ... fields
    }).index("by_user_and_channel", ["userId", "channel"]),
    ```
    
    <Tip>
      Field order matters! Put the most selective field first.
    </Tip>
  </Tab>
</Tabs>

### Index Usage Analysis

<AccordionGroup>
  <Accordion title="Unused Indexes">
    Identifies indexes that are never queried:
    
    - **Index name**: by_old_field
    - **Created**: 30 days ago
    - **Usage**: 0 queries in last 7 days
    - **Recommendation**: Consider removing
    - **Impact**: Reduces write overhead
    
    <Warning>
      Verify index is truly unused before removing. Check all environments.
    </Warning>
  </Accordion>
  
  <Accordion title="Redundant Indexes">
    Finds overlapping indexes:
    
    ```typescript
    // Redundant indexes detected:
    .index("by_user", ["userId"])          // ← Redundant
    .index("by_user_created", ["userId", "_creationTime"])  // ← Covers above
    
    // Recommendation: Remove "by_user" index
    // "by_user_created" can serve both queries
    ```
  </Accordion>
  
  <Accordion title="Index Selectivity">
    Measures how effectively indexes narrow results:
    
    - **High selectivity** (good): Index narrows to less than 1% of rows
    - **Medium selectivity** (okay): 1-10% of rows
    - **Low selectivity** (poor): >10% of rows
    
    Low selectivity indexes may not improve performance.
  </Accordion>
</AccordionGroup>

### Optimization Suggestions

<Tabs>
  <Tab title="Add Pagination">
    **Limit Result Sets**
    
    ```typescript
    // ❌ Before: Fetching all messages
    const messages = await ctx.db.query("messages")
      .filter(q => q.eq(q.field("channel"), channel))
      .collect();  // Could return thousands!
    
    // ✅ After: Paginated query
    const messages = await ctx.db.query("messages")
      .withIndex("by_channel", q => q.eq("channel", channel))
      .order("desc")
      .take(50);  // Only 50 most recent
    ```
    
    **Impact**: Reduces query time by 90%+ for large tables
  </Tab>
  
  <Tab title="Use Indexes">
    **Leverage Index Ordering**
    
    ```typescript
    // ❌ Before: Sorting without index
    const messages = await ctx.db.query("messages")
      .collect()
      .sort((a, b) => b._creationTime - a._creationTime);  // In-memory sort
    
    // ✅ After: Index-based ordering
    const messages = await ctx.db.query("messages")
      .withIndex("by_creation_time")  // Uses index order
      .order("desc")
      .take(50);
    ```
    
    **Impact**: Eliminates expensive in-memory sorting
  </Tab>
  
  <Tab title="Batch Operations">
    **Avoid N+1 Queries**
    
    ```typescript
    // ❌ Before: N+1 pattern
    const messages = await ctx.db.query("messages").collect();
    for (const message of messages) {
      const user = await ctx.db.get(message.userId);  // N queries!
      // ...
    }
    
    // ✅ After: Batch load
    const messages = await ctx.db.query("messages").collect();
    const userIds = [...new Set(messages.map(m => m.userId))];
    const users = await Promise.all(
      userIds.map(id => ctx.db.get(id))
    );  // Parallel fetch
    const userMap = new Map(users.map(u => [u._id, u]));
    ```
    
    **Impact**: 10x-100x faster for large datasets
  </Tab>
  
  <Tab title="Reduce Over-Fetching">
    **Select Only Needed Fields**
    
    ```typescript
    // ❌ Before: Fetching entire documents
    const users = await ctx.db.query("users")
      .collect();  // All fields, including large ones
    
    // ✅ After: Project specific fields (coming soon)
    // For now, filter in application code:
    const users = (await ctx.db.query("users").collect())
      .map(u => ({ _id: u._id, name: u.name }));  // Only needed fields
    ```
    
    **Impact**: Reduces bandwidth and memory usage
  </Tab>
</Tabs>

## Schema Design Recommendations

### Relationship Optimization

<AccordionGroup>
  <Accordion title="Denormalization Opportunities">
    Suggests when to denormalize for performance:
    
    ```typescript
    // Detected pattern: Frequently joining users for author name
    
    // Current schema:
    messages: defineTable({
      userId: v.id("users"),
      text: v.string(),
    })
    
    // Recommended: Add denormalized field
    messages: defineTable({
      userId: v.id("users"),
      authorName: v.string(),  // Denormalized
      text: v.string(),
    })
    
    // Trade-off: Faster reads, slightly more complex writes
    ```
  </Accordion>
  
  <Accordion title="Normalization Suggestions">
    Identifies over-denormalization:
    
    - Duplicate data across many tables
    - Large embedded objects
    - Inconsistency risks
    - Suggests extraction to separate table
  </Accordion>
  
  <Accordion title="Document Size Warnings">
    Alerts for large documents:
    
    - Documents >16KB (approaching limit)
    - Embedded arrays with many items
    - Large string fields
    - Suggests splitting into multiple documents
  </Accordion>
</AccordionGroup>

### Field Type Optimization

```typescript
// Suboptimal type detected:
messages: defineTable({
  timestamp: v.string(),  // ⚠️ Storing number as string
})

// Recommended:
messages: defineTable({
  timestamp: v.number(),  // ✅ Proper numeric type
})
// Benefits: Better indexing, range queries, less storage
```

## Troubleshooting

<AccordionGroup>
  <Accordion title="No recommendations shown">
    - Wait for enough data collection (24-48 hours)
    - Verify deployment connection
    - Check if application has traffic
    - Try refreshing the analysis
  </Accordion>
  
  <Accordion title="Can't apply auto-fix">
    - Ensure schema.ts file is writable
    - Check file isn't open in editor
    - Verify correct project directory
    - Try manual fix if auto-fix fails
  </Accordion>
  
  <Accordion title="Improvement not seen after fix">
    - Wait for cache to clear (5-10 minutes)
    - Verify index was created
    - Check query is using new index
    - Monitor for 15-30 minutes
  </Accordion>
</AccordionGroup>
